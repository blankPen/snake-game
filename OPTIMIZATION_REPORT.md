## 4. 开发流程规范建议

### 4.1 分支策略

```
main                    # 主分支，稳定版本
  ├── feature/v1.0-config     # Alpha: 配置中心
  ├── feature/v1.0-snake      # Alpha: 蛇实体
  ├── feature/v1.0-food       # Beta: 食物系统
  ├── feature/v1.0-renderer   # Alpha: 渲染引擎
  ├── feature/v1.0-input      # Beta: 输入处理器
  └── feature/v1.0-game       # Alpha: 游戏控制器
```

**分支命名规则：**
- `feature/v{version}-{module}` - 功能开发分支
- `bugfix/v{version}-{issue}` - Bug 修复分支
- `hotfix/v{version}-{fix}` - 紧急修复分支

### 4.2 提交规范

**Conventional Commits：**
```
feat: 新增功能
fix: 修复 Bug
docs: 文档更新
style: 代码格式（不影响功能）
refactor: 重构代码
test: 测试相关
chore: 构建或工具链
```

**示例：**
```bash
git commit -m "feat(snake): 实现蛇移动和转向逻辑"
git commit -m "fix(renderer): 修复网格线绘制错位问题"
git commit -m "docs: 更新 README 添加安装说明"
```

### 4.3 PR 提交规范

**PR 标题格式：**
```
[模块] 简短描述
```

**PR 模板：**
```markdown
## 变更类型
- [ ] 新增功能
- [ ] Bug 修复
- [ ] 文档更新
- [ ] 代码重构

## 变更内容
简要描述本次变更的内容

## 关联 Issue
Closes #xxx

## 测试情况
- [ ] 本地测试通过
- [ ] 单元测试通过
- [ ] 手动测试场景：xxx

## 备注
其他需要说明的事项
```
